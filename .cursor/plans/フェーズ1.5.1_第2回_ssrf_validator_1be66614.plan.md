# 第2回：SSRF 防衛バリデーター（開発計画・承認用）

## 1. 制約の確認

### 今回のスコープで「やること」

- 新規クラス **[SsrfValidator.java](c:\cursor\project\geo-analytics\src\main\java\com\geo\analytics\infrastructure\security\SsrfValidator.java)** を追加する。
- Spring の **`@Component`** を付与し、**`public void validate(String targetUrl)`** のみを公開 API（内部は private メソッド分割可）。
- 検証は要件どおり **次の順**で実行する: (1) null / 空文字 → (2) URI 構文 → (3) スキーム `http` / `https` のみ → (4) ホストがローカル名・禁止 IPv4 / 禁止 IPv6 ループバックに該当しないこと。
- いずれかのルール違反時は、**[ScrapingException](c:\cursor\project\geo-analytics\src\main\java\com\geo\analytics\domain\exception\ScrapingException.java)** を **`new ScrapingException("…明確な英語理由…")`** のように **メッセージで理由を明示**してスローする（例: `"Invalid scheme"`, `"Private IP is not allowed"`, `"Host not allowed"` など、ルールごとに固定文言でよい）。
- 配置パッケージは既存の security コンポーネント（[JwtAuthenticationFilter](c:\cursor\project\geo-analytics\src\main\java\com\geo\analytics\infrastructure\security\JwtAuthenticationFilter.java) 等）と同様 **`com.geo.analytics.infrastructure.security`** とする。

### 今回「絶対にやらないこと」

- **Jsoup・`HttpURLConnection`・`HttpClient`・ソケット接続など、実際の HTTP 取得・HTML パース**を書かない。
- **`InetAddress#getByName` / `getAllByName`、`InetSocketAddress` の名前解決**など、**外部問い合わせや OS レゾルバを踏む API**を使わない。
- **このフェーズでは** `UrlContentFetcher` の実装や、バリデータをそこへ組み込む配線まで**広げない**（ユーザー指示の対象ファイルは `SsrfValidator` のみと解釈。必要なら第3回以降で接続）。

### 設計上の割り切り（承認時の共有）

- **文字列・URI 構文のみ**では、「合法的な公衆ホスト名が実際にはプライベート IP を指す」**DNS リバインディング**までは防げない。今回は指示どおり **DNS を禁止**する代償として、そのリスクは **Fetcher 側の別層防御（将来的にプロキシ固定・resolve 方針）**に委ねる前提とする。

## 2. ロジック設計

### URL のパースに使う標準 API

- **`java.net.URI`** を使用する。`new URI(targetUrl)`（または同等のコンストラクタ）でパースし、`URISyntaxException` を捕捉して **「URI として不正」**として `ScrapingException` にラップしてもよい（メッセージ例: `"Invalid URL syntax"`）。
- パース後に利用するのは主に次の **`String` のみ**（ここでも名前解決は発生しない）:
  - **`getScheme()`** — 小文字化して `http` / `https` と完全一致するか判定（`file:`, `jar:`, `javascript:` 等は拒否）。
  - **`getHost()`** — 権威あるホスト部分。`null`（不透明 URI、scheme だけ、欠落など）は **ホスト不明として拒否**。
- **`getRawAuthority()` / `getUserInfo()`** 等: ユーザー情報付き URL（`http://user@host/...`）は SSRF の周辺で紛らわしいが、**ホスト判定は `getHost()` に統一**し、将来フェーズで「userinfo 禁止」を足す余地は残す（今回は要件に無ければ最小限: `getHost()` が正しくホストを返せない形は構文エラーに近い扱い）。

### ホスト名・IP 制限の判定方針（正規表現・文字列操作）

**共通前処理**

- `host = uri.getHost()`。**大文字小文字はホスト比較で正規化**（`toLowerCase(Locale.ROOT)`）してからパターン適用（`LocalHost` 等の抑止）。
- **IPv6**: `URI` のホストは **`[` `]` なし**で返る場合と、入力により **`[` `]` 付き**が残る表記差があるため、判定前に **先頭・末尾の `[` `]` を除去**してから 1 本の「正規化ホスト文字列」にまとめる。

**A. ホスト名（文字列）**

- **`localhost`**（および一般的な **`127.0.0.1` 以外の名前でのループバック名**として要件があるもののみ）: 正規表現または **`equals("localhost")`**（正規化後）で拒否。

**B. IPv4（ドット十進）**

- ホストが **IPv4 らしいパターン**（`^\d{1,3}(\.\d{1,3}){3}$` 程度でプリチェック）のとき、次を **順に／または正規表現で**拒否:
  - **`127.*`**（ループバック全体）
  - **`10.*`**
  - **`192.168.*`**
  - **`172.16.*`〜`172.31.*`** — 例: `^172\.(1[6-9]|2\d|3[0-1])\..*`（サブネット全体）
  - **`169.254.*`**（リンクローカル / メタデータ系の典型帯）
- **オクテットが 0–255 外**の表記（`999.999.…`）は URI としてはあり得るため、**数値範囲チェック**で弾くか、不正オクテットは **「不正 IP 形式」**で拒否（実装時は一貫したメッセージにする）。
- 要件外だが SSRF 対策として **`0.0.0.0`** や **`::`** に相当する明白なゼロアドレスが来た場合の扱いは、実装時に **プライベート／非ルーティング相当としてブロック**するかアーキテクト判断で追加可能（計画では「要件の禁止帯を満たしたうえで、明らかな危険は同じ `ScrapingException` で拒否」を推奨）。

**C. IPv6 ループバック（要件の「等」を満たす範囲）**

- ブラケット除去後の文字列に対し、**大文字小文字無視**で次をカバー:
  - 圧縮形 **`::1`**
  - フル形 **`0:0:0:0:0:0:0:1`**
  - 中間ゼロ省略の別表記（実装では **小さな正規表現セット** または **ゼロランの正規化**で `::1` 系に寄せる方針を取る）
- **`::ffff:127.0.0.1`（IPv4-mapped ループバック）** は、ホスト文字列に `127.` が含まれる形で現れるため、**IPv4 部分のブロック**または **専用パターン**で拒否する。

**D. 明示リストにないが「明らかに危険」な表記**

- 空ホスト、**全角ドット**混入など、信頼できない入力は **早期に構文／ホスト不正**で拒否。

### テスト（実装フェーズでの推奨）

- 今回の指示に **テストファイルは含まれていない**が、承認後の実装では **JUnit で許可 URL / 拒否 URL の表駆動テスト**を追加することを推奨（`InetAddress` 不要）。テスト追加がスコープ外なら省略。
