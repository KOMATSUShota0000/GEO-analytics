# ADR-009: 競合ベンチマークの正直化（参考基準点プロファイル伝播）＋ Gemini空コンテンツ・ガード

## 日付
2026-05-25

## 状況

オーナー実機検証（Sprint10.7）で次候補として記録された2件を処理した。

- **B-1（競合URL未表示）**: 実競合が不足するとき、システムは合成競合（参考基準点）を補充する。
  合成競合は実在しないため URL を null にしている。さらに調査した結果、**合成競合の名称・理由・合成フラグは、
  フロントへ届く前の段階（`SelectedCompetitor` → 競合URL文字列リストへの平坦化 `competitorUrlsFromSelected`）で
  すべて破棄されていた**。その結果フロントは URL 文字列のみを受け取り、合成競合は空文字 `""` となって
  「空の `<a href="">` が3件並ぶ」状態（React の key 重複も発生）になっていた。これが「競合URL未表示」の真因。
- **C-1（Gemini空コンテンツ400）**: 空の `contents` を Gemini に渡すと `400 INVALID_ARGUMENT` でログが汚れる
  （UI影響なし）。

## 決定

### B-1: 競合プロファイルを非破壊で永続化・露出

- ドメインモデル `CompetitorProfile(name, websiteUrl, synthetic)` を新設。
- `ProjectEntity` に JSONB 列 `competitor_profiles` を追加（Flyway V127）。既存の `project_competitors`
  （URL文字列・ElementCollection）は**一切変更しない**（非破壊）。前例 `minority_reports`（JSONB）に倣う。
- ジョブのハイブリッド競合確定時（`JobQuerySubmissionService`）に、URL文字列リストの保存に加えて
  `SelectedCompetitor` から `CompetitorProfile` リストを生成して保存する。合成競合は URL を null のまま保持。
- `JobProjectResponse` に `competitorProfiles`（任意・追加フィールド）を露出。既存フィールドは不変。
- フロント（`JobAnalysisPage` / `ReportPrintPage`）は、プロファイルがあれば実競合（リンク表示）と
  参考基準点（「参考基準点・実競合ではない」バッジ・URL非表示）を出し分ける。全件合成のときは
  「実競合が十分に取得できなかったため参考基準点を表示しています」の注記を出す。
  プロファイル未保持の旧ジョブは、空URLを除外したうえで従来のURL表示にフォールバックする。

### C-1: 発生源で空コンテンツ呼び出しを止める

- `GapDiagnosisGeminiClient.analyze(userContent)` の入口で、`userContent` が null/空白なら
  Gemini を呼ばず中立フォールバック（理由なし＋汎用アクション3件）を返し `WARN` を1行記録する。

## 理由

- **B-1**: 「合成だと分かるバッジを足す」には、合成フラグと意味のある名称がフロントまで届く必要がある。
  既存経路はそれらを破棄していたため、フロントだけでは実現不可能だった。追加列＋追加フィールドの
  非破壊設計とすることで、既存の競合URL依存箇所（円グラフ・印刷）を退行させずに実現できる。
  捨てていた良質な動的 reasoning／参照ティア名も活用でき、組織の品質基準「スタブやモックで誤魔化さない」に合致する。
- **C-1**: 発生源で止めることで、無駄な LLM 呼び出し（クレジット消費）とエラーログ汚染の双方を排除できる。
  応急処置（ログ降格）では無駄な呼び出しが残る。

## 結果

- 競合表示が「実競合 vs 参考基準点」を正直に区別。空URLリンク／key重複が解消。レポート印刷にも反映。
- 追加APIフィールドを持たない過去ジョブはフロントが従来表示へフォールバックし破綻しない（後方互換）。
- C-1 は空コンテンツの一クラスのバグを発生源で塞いだ。

### 正直な但し書き（C-1のスコープ）

静的監査の結果、**配線済みの Gemini 呼び出し（GBVS検証・競合推論・巨大フィルタ・バッチ本体・ギャップ分析バッチ・
感情アラート）は、いずれも入力をガード済みか、非空のプロンプトテンプレートで本文を包んでおり、空 contents を送らない**。
生の未ガード入力を渡し得る唯一の経路が `GapDiagnosisGeminiClient.analyze` だが、これは現状どこからも**未配線**である。
したがって、オーナーが観測した過去の `400` ログの**正確な発生源は静的解析では一意に特定できなかった**。
本ガードは「空 contents → 400」というクラスの再発防止（および当該クライアントが配線された際の予防）として正しい修正であるが、
**過去ログの完全な根絶を検証済みとは主張しない**。次回の実機解析でログを確認し、別経路で 400 が再発する場合は
当該経路に同種ガードを追加する方針とする。
