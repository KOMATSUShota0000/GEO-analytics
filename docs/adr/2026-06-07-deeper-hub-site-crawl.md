# ADR-036: ハブ型サイトの取りこぼし対策 — 巡回ページ数と結合上限の拡張

## 日付
2026-06-07

## 状況
多くのサイトは**トップページが薄いハブ型**で、実コンテンツ（こだわり/メニュー/店舗/会社概要/採用 等）が複数のサブページに分散する。実例 `https://fbih.jp/`（おにぎりこんが）はトップが導線＋お知らせ抜粋のみで、8サブページに内容が分散していた。

従来のクロールは **入口＋追従2＝最大3ページ**（`CorePageLinkScorer.MAX_FOLLOW=2`、`SmartDomainCrawlService` の上限3）で、結合監査テキストは **15,000文字**上限（`DomainDeepAuditContext.MERGED_TOTAL_MAX_CHARS`）。これだとハブ型サイトでメニュー（料金）・店舗（地域）・採用（経歴）等を取りこぼし、ルーブリックの該当項目が過小評価＝スコアとエビデンスが実態より低く出ていた。

費用試算: ルーブリック評価は Gemini 2.5 Flash・1解析1回・入力≤上限文字。Flash入力は安価（概算 $0.1〜0.3/1Mトークン）で、上限を15k→24kにしても**増分は概ね1円未満/解析**。巡回ページ増の追加コストはHTTP取得のみ（LLM非使用）。＝費用は制約にならない（案B寄りが妥当）。

## 決定
ハブ型サイトを高精度に解析するため、巡回深さと結合上限を拡張する（定数変更の低リスクな第一歩）。

- `CorePageLinkScorer.MAX_FOLLOW`: 2 → **5**（重要度上位5サブページまで追従）。
- `SmartDomainCrawlService`: 総巡回ページ上限 3 → **6**（定数 `MAX_AUDIT_PAGES`。入口＋最大5サブ）。
- `DomainDeepAuditContext.MERGED_TOTAL_MAX_CHARS`: 15,000 → **24,000**（増えたページの本文が収まるよう拡張）。

全プラン一律で適用（まず全ユーザーの精度を底上げ）。

## 理由
- ハブ型が大半という実態に対し、3ページ上限が信頼性のボトルネックだった（fbih.jp で実証）。
- 費用は1円未満/解析の増分で制約にならない（Flash・1回・上限頭打ち）。精度を取りに行くべき。
- リンク採点（`CorePageLinkScorer`）が既に重要ページを選別するため、ページ数を増やしても無関係ページを掴みにくい。
- 定数変更のみでロジック不変＝低リスク。per-page要点抽出の高度化・プラン別N（上位プランほど深く）は後続の最適化。

## 結果
- 入口＋上位5サブページ（計6）を巡回し、結合24kでルーブリック監査・技術評価にかける。ハブ型サイトのメニュー/店舗/採用等を拾えるようになり、スコア・エビデンス（ADR-035）の精度が向上する見込み。
- トレードオフ: クロール時間が増える（ページ数比例のHTTP取得＝レイテンシ微増）。LLMコストはほぼ不変。
- 申し送り（pending-implementations）: ①per-page要点抽出（料金/地域/経歴を優先して24k予算に賢く詰める）②プラン別の巡回ページ数N（アップセル）③クエリ数1問題（別軸の信頼性）。
